# v1 后台布局、菜单与 RBAC 设计

## 1. 背景

`admin-backend-3/apps/page` 是当前项目的正式前端。第一版目标不是复刻 Vben 的完整工程封装，而是在普通 Vue 3 + Element Plus 项目中拆解并实现 Vben 风格的后台管理前端。

当前前端需要同时参考三个来源：

| 来源                            | 作用                                                        |
| ------------------------------- | ----------------------------------------------------------- |
| `admin-backend/frontend`        | 旧业务来源，只保留数据、接口、业务流程和权限逻辑            |
| `admin-backend-2/apps/page-new` | UI 来源，参考后台框架、布局气质、表单、表格和右上角工具区   |
| `admin-backend-3/apps/page`     | 当前实现目标，保持普通前后端项目结构，不引入 `@vben` 高封装 |

### 1.1 UI 迁移硬规则

后续所有旧页面迁移都按下面规则处理：

```text
旧项目提供业务内容，不提供最终 UI。
Vben 提供 UI 目标，不直接提供高封装实现。
当前项目用普通 Vue + Element Plus 重新拆解实现。
```

因此，“保留旧项目”只表示保留旧项目里的数据结构、接口调用、业务流程、权限逻辑和页面能力范围，不表示保留旧项目的页面样式、布局结构、交互形态或组件写法。

## 3. 新菜单设计

新项目不照搬旧菜单，也不把旧功能硬塞进参考项目的两个菜单中。`概览` 和 `系统管理` 保留为基础后台菜单，旧业务功能按业务域重新组织。

```text
概览
  - 数据概览 / 工作台

数据统计
  - 每日数据统计
  - 历史数据查询

数据报告
  - 报告编辑
  - 数据报表
  - 结算单

Excel管理
  - 文件上传
  - 文件列表
  - 数据查看（隐藏路由）

API 管理
  - API Key 管理
  - 我的 API Key
  - API 文档

系统管理
  - 用户管理
  - 用户数据
  - 角色管理
  - 权限管理
  - 菜单管理

异常页面
  - 403 / 404 / 500（隐藏路由）
```

### 菜单设计原则

| 原则           | 说明                                                   |
| -------------- | ------------------------------------------------------ |
| 业务域优先     | 按数据统计、数据报告、Excel、API、系统管理分组         |
| 保留学习价值   | 旧项目中有真实流程、接口、数据和权限关系的能力优先保留 |
| 去掉低价值入口 | `welcome` 页面不迁移，用新 `概览` 代替                 |
| 隐藏详情页     | 详情页、错误页保留路由能力，但不进入侧边栏             |
| UI 统一重做    | 不沿用旧项目前端视觉，统一拆解 Vben 的页面形态实现     |
| 不引入高封装   | 不直接复用 `@vben` 布局包，用普通组件拆解实现          |

### RBAC 第一版边界

| 能力     | 第一版处理                                    |
| -------- | --------------------------------------------- |
| 菜单权限 | 根据登录用户角色和管理员层级过滤侧边栏        |
| 页面权限 | 路由守卫检查 `requiresAuth`、角色和管理员层级 |
| 按钮权限 | 先在关键页面以显式判断实现，后续抽象权限按钮  |
| 数据权限 | 保留后端控制，前端只表达入口权限              |
| 动态路由 | 暂不做，等 RBAC 数据模型稳定后再接入          |

## 6. 页面迁移策略

第一版不直接把旧项目页面大文件整体搬入当前项目，也不迁移旧项目的页面 UI。

原因：

1. 旧页面中混合了业务逻辑、接口调用、复杂表格、样式和历史封装。
2. 当前项目目标是复习和学习，不应一次性引入难以拆解的大组件。
3. 菜单、路由、RBAC、布局是后台项目主线，应先稳定。
4. 当前 UI 目标已经明确为 Vben 风格，旧 UI 只能作为反例或业务核对材料。

推荐迁移顺序：

```text
1. 后台布局
2. 菜单和路由
3. RBAC 过滤
4. 概览页
5. 用户管理
6. 数据统计
7. Excel 管理
8. 数据报告
9. API 管理
10. 错误页和隐藏详情页
```

每个业务页面迁移时都应拆解为：

```text
页面视图
业务组件
请求 API
状态 / 查询
类型定义
测试
文档记录
```

每个页面的拆解顺序应固定为：

```text
1. 从旧项目提取数据字段、接口、权限判断和业务动作
2. 从 Vben 参考页提取页面布局、表单密度、表格工具栏和交互方式
3. 在当前项目中用普通 Vue 组件重新实现
4. 为纯业务逻辑补测试，为页面实现补子文档
```

## 7. 第一版不做的内容

| 不做项                   | 原因                                  |
| ------------------------ | ------------------------------------- |
| 直接复制 `@vben` 布局包  | 会引入高封装，降低学习粒度            |
| 直接复制旧项目所有大页面 | 会把旧 UI 和历史复杂度一次性迁移过来  |
| 沿用旧项目前端 UI        | 当前 UI 目标是 Vben 风格，不能混用    |
| 后端动态菜单             | RBAC 模型还需要先在前端静态版本中跑通 |
| 偏好设置抽屉             | 当前不是核心后台能力                  |
| 顶部搜索 / 全屏 / 多语言 | 先聚焦后台主流程                      |
| 欢迎页                   | 已确认低价值，用新概览替代            |

## 9. 当前结论

第一版后台框架应该是：

```text
左侧：业务菜单 + RBAC 过滤
顶部：面包屑 + 通知 + 主题 + 用户
内容区：Element Plus 风格业务页面
路由：静态路由 + 前端权限过滤
页面：旧业务能力保留，旧 UI 不保留，低价值欢迎页忽略，详情页隐藏
```

这个方向符合当前项目目标：用可理解的普通前端工程方式复习后台管理全域知识点，同时保留旧项目的核心业务能力，并把最终 UI 统一到 Vben 风格。

## 10. 当前实施记录

本阶段已经把文档中的后台框架规划落到 `apps/page` 代码中。

### 10.1 路由落地

业务路由现在统一挂载在主布局下：

```text
/
  MainLayout
    /dashboard
    /statistics/*
    /reports/*
    /excel/*
    /api/*
    /system/*
```

这样做的目的：

| 目标         | 说明                                           |
| ------------ | ---------------------------------------------- |
| 布局统一     | 登录页之外的后台页面都共用侧边栏、顶部栏       |
| 菜单自动生成 | 侧边栏仍然从 `appRoutes` 的 meta 信息派生      |
| 权限不分裂   | 页面访问和菜单可见范围继续共用同一份 RBAC meta |
| 后续可扩展   | 之后接入后端动态路由时，替换路由来源即可       |
