# 路由系统：静态路由、守卫与 RBAC 菜单

## 1. 当前目标

`apps/page` 采用"静态路由 + `meta.permission` 权限码判定 + 守卫/菜单同源过滤"的方式实现后台管理路由权限。

这个方案属于传统后台管理中常见的 RBAC 前端落地方式：页面路由通过 `meta` 声明需要的权限码或角色，路由守卫负责拦截，菜单根据同一套路由权限过滤。

它的重点不是"前端决定最终权限"，而是"前端表达入口权限和交互权限"。真正的数据访问、接口写入、数据范围控制仍然必须由后端校验（`requirePermission` 中间件）。前端任何路由方案都拦不住直接 curl 接口，所以前端权限本质上是 UX 层。

这一层解决的问题：

| 问题           | 当前处理方式                                                         |
| -------------- | -------------------------------------------------------------------- |
| 页面在哪里声明 | 在 `src/router/routes.ts` 中集中声明                                 |
| 菜单从哪里来   | 从 `appRoutes` 的路由 `meta` 派生                                    |
| 未登录如何拦截 | 全局路由守卫跳转 `/login`，并保留 `redirect`                         |
| 无权限如何处理 | 全局路由守卫跳转 `/403`                                              |
| 权限如何表达   | `meta.permission` 权限码（后端下发）；未声明权限码的路由登录即可访问 |
| 按钮级权限     | `v-permission` 指令按权限码控制按钮显隐                              |

## 2. 源码位置

| 文件                                   | 作用                                                     |
| -------------------------------------- | -------------------------------------------------------- |
| `src/router/routes.ts`                 | 声明登录页、业务页、错误页、菜单标题、图标、角色、权限码 |
| `src/router/types.ts`                  | 扩展 Vue Router 的 `RouteMeta` 类型                      |
| `src/router/guards.ts`                 | 注册全局路由守卫                                         |
| `src/router/access.ts`                 | 判断用户是否能访问当前路由（守卫与菜单共用）             |
| `src/router/menu.ts`                   | 根据路由和用户权限生成侧边栏菜单数据                     |
| `src/router/redirect.ts`               | 登录后跳转目标解析（防开放重定向）                       |
| `src/auth/permissions.ts`              | 权限码判定（`hasPermission`）                            |
| `src/directives/permission.ts`         | `v-permission` 按钮级权限指令                            |
| `src/stores/auth.ts`                   | 保存当前 token、用户快照（含 `permissions` 权限码）      |
| `src/views/error/*.vue`                | 403 / 404 / 500 错误页                                   |
| `src/views/FeaturePlaceholderView.vue` | 业务页面迁移前的路由占位页                               |

权限码常量来自 `contracts/admin-api/src/permissions.ts`，前后端共享同一份，避免字符串拼错。

## 3. 路由元信息设计

当前路由 `meta` 主要使用这些字段：

| 字段           | 说明                                                                              |
| -------------- | --------------------------------------------------------------------------------- |
| `title`        | 页面标题，也是菜单标题来源                                                        |
| `icon`         | 菜单图标标识，后续侧边栏组件再映射到 Element 图标                                 |
| `order`        | 同级菜单排序                                                                      |
| `requiresAuth` | 是否需要登录                                                                      |
| `guestOnly`    | 已登录用户访问时自动跳回后台                                                      |
| `hideInMenu`   | 是否隐藏在侧边栏中                                                                |
| `activeMenu`   | 隐藏详情页用于高亮对应菜单                                                        |
| `permission`   | RBAC 权限码：声明后由后端下发的 `user.permissions` 判定；未声明的路由登录即可访问 |

示例（`/system/users`、`/system/roles` 是 `meta.permission` 样板）：

```ts
import { PERMISSION_CODES } from "@admin-backend-3/admin-api-contract/permissions";

{
  path: "/system/users",
  name: "SystemUsers",
  meta: {
    title: "用户管理",
    // 由后端实时解析的权限码判定访问
    permission: PERMISSION_CODES.systemUserView,
  },
}
```

判定规则（`src/router/access.ts`）：声明了 `meta.permission` 的路由用权限码判定（细粒度，改权限后刷新令牌即生效）；未声明的路由登录即可访问（概览、公共组件等布局级路由无需细粒度权限码）。菜单生成复用同一判定，无权限的菜单项自动隐藏。

`meta.roles` 角色回落已随 `users.role / users.admin_level` 字段删除（迁移 022）一并移除：路由与菜单显隐一律走权限码，角色码只用于展示和 super 专属规则。

## 4. RBAC 数据来源

后端在登录/刷新响应里实时解析下发两个数组（不进 JWT，改角色或权限后刷新令牌即生效）：

```text
roles         # 角色码列表，按 user_roles -> roles 解析；仅用于展示与 super 专属规则（stores/auth.ts 的 isSuper）
permissions   # 权限码列表，按 user_roles -> roles -> role_menus -> menus.auth_code 解析；路由/菜单/按钮显隐的唯一判定依据
```

`users` 表本身不存角色字段（`role` / `admin_level` 已在迁移 022 删除），`user_roles` 绑定表是角色归属的唯一数据源。前端不再做"后端字段 -> 前端访问角色"的映射（原 `src/auth/rbac.ts` 已删除）。

## 5. 守卫流程

当前全局守卫顺序（`src/router/guards.ts`）：

```text
进入路由
  -> 若内存中没有 access token，先通过 HttpOnly Cookie 调用刷新接口恢复会话（await）
  -> 已登录访问登录页：跳转到 redirect 或 /dashboard
  -> 未登录访问受保护页面：跳转 /login?redirect=目标地址
  -> 已登录但权限不满足：跳转 /403?from=目标地址
  -> 访问 / ：跳转 /dashboard
  -> 其它情况放行
```

几个实现细节对应 vue-router 4 的推荐做法：

| 细节                               | 说明                                         |
| ---------------------------------- | -------------------------------------------- |
| 守卫用返回值导航                   | 不使用已不推荐的 `next()` 回调               |
| 异步会话恢复 `await`               | 不会在会话未恢复时放行或误判未登录           |
| 跳转前判断目标是否就是 Login / 403 | 防止重定向死循环                             |
| `to.matched.every()` 逐层校验      | 父路由无权限时子路由也进不去，不只查叶子路由 |

## 6. 菜单派生

菜单不再手写一份独立配置，而是从 `appRoutes` 派生：

```text
路由配置
  -> 按用户权限过滤（与守卫共用 access.ts 的同一判定）
  -> 去掉 hideInMenu
  -> 按 order 排序
  -> 目录路由子项全被权限过滤后整体隐藏（避免空目录）
  -> 生成侧边栏菜单
```

这样可以保证：

| 好处             | 说明                                            |
| ---------------- | ----------------------------------------------- |
| 页面和菜单不分裂 | 新增页面时只维护路由元信息                      |
| 权限判断统一     | 路由守卫和菜单过滤都使用同一套判定函数          |
| 隐藏路由清晰     | 详情页、错误页可以保留路由能力，但不进入菜单    |
| 方便后续替换     | 后端动态路由稳定后，可以替换 `appRoutes` 数据源 |

## 7. 后台路由方案全景

后台管理项目的路由权限方案，按"路由从哪来 + 权限怎么控"可以分为六类。前五类是权限控制维度，第六类是路由来源维度（可与前五类正交组合）。

### 7.1 手写菜单 + 手写路由

菜单和路由分别写成两份独立配置，权限判断散落在各处。

```text
sidebar-menu.ts（手写菜单树）
routes.ts（手写路由表）
各页面/组件里自行 if (role === 'admin') ...
```

| 项目     | 说明                                                                     |
| -------- | ------------------------------------------------------------------------ |
| 优点     | 初期最快，零抽象成本                                                     |
| 缺点     | 菜单、路由、权限三处分裂；新增页面要改多处；权限判断无单一入口，必然腐化 |
| 适用场景 | 原型、一次性内部工具、页面数 < 5 的小后台                                |
| 代表实现 | 无（这是所有项目的"默认起点"，不是方案）                                 |
| 当前结论 | 不采用                                                                   |

### 7.2 静态全量注册 + `meta` 声明 + 守卫过滤（当前方案）

启动时把全部业务路由注册进 Vue Router，权限信息声明在 `meta` 上；全局守卫拦截无权限访问，菜单从同一套 `meta` 过滤派生。

```text
启动：createRouter(全量 routes)
守卫：beforeEach -> canAccessMatchedRoute(to, user) -> 放行 / 403
菜单：appRoutes -> 同一判定过滤 -> 侧边栏
```

| 项目       | 说明                                                                                                                             |
| ---------- | -------------------------------------------------------------------------------------------------------------------------------- |
| 优点       | 最易理解和测试；路由表所见即所得，调试直观；无动态注入的时序问题；菜单/守卫天然同源                                              |
| 缺点       | 所有路由声明进 bundle（懒加载下泄露的只是路径名）；无权限访问返回 403 而非 404，暴露"该路径存在"；权限粒度取决于 meta 声明的维护 |
| 适用场景   | 中小后台、学习项目、权限模型仍在演进的项目                                                                                       |
| 代表实现   | 大量中小项目；本项目                                                                                                             |
| 关键实现点 | 守卫和菜单必须共用同一判定函数；`to.matched` 逐层校验；权限码用常量不用裸字符串                                                  |
| 当前结论   | **当前采用**                                                                                                                     |

### 7.3 静态路由表 + 登录后过滤 + `addRoute` 动态注入

完整路由表仍由前端维护，但启动时只注册基础路由（登录、404 等）。登录拿到用户信息后，按角色/权限码过滤出可访问路由，再 `router.addRoute()` 动态注入。

```text
启动：createRouter(基础 routes)
登录成功：filterRoutes(allRoutes, user) -> addRoute 逐条注入 -> 生成菜单
登出/切换账号：removeRoute 重置 -> 重新过滤注入
刷新页面：恢复会话 -> 重新过滤注入 -> 再放行首次导航
```

| 项目       | 说明                                                                                                                                         |
| ---------- | -------------------------------------------------------------------------------------------------------------------------------------------- |
| 优点       | 无权限路由不存在于 router，直接 404（不暴露路径存在性）；router 实例干净；菜单来自最终可访问路由                                             |
| 缺点       | 刷新时"先注入再导航"的时序要处理好（常见白屏/闪 404 坑）；切换账号要正确 removeRoute；守卫里需要"路由是否已就绪"的状态位；整体复杂度明显上升 |
| 适用场景   | 中大型前端项目、通用后台模板、对 404 隐藏有要求的系统                                                                                        |
| 代表实现   | vue-element-admin、vben `frontend` 模式（`generateRoutesByFrontend()` + `meta.authority`）                                                   |
| 关键实现点 | 动态注入后首次导航要 `next({...to, replace: true})` 重进一次（vue-router 3）或确保注入完成再放行（vue-router 4）                             |
| 当前结论   | 后续可升级                                                                                                                                   |

### 7.4 后端动态路由

菜单和路由结构由后端接口返回（通常来自菜单表），前端只负责把后端的组件字符串映射为真实页面组件。

```text
登录成功：GET /api/menus -> [{ path, component: "views/system/users/index", meta }]
前端：pageMap["views/system/users/index"] -> 真实组件 -> addRoute 注入
```

| 项目       | 说明                                                                                                                             |
| ---------- | -------------------------------------------------------------------------------------------------------------------------------- |
| 优点       | 权限配置集中在后端；改菜单/权限不需要前端发版；运营或管理员可通过后台配置入口；多租户天然支持                                    |
| 缺点       | 需要稳定的菜单表、权限表、组件路径协议；组件字符串映射失败要有兜底；前后端耦合一份协议，联调成本高；前端本地开发依赖后端菜单数据 |
| 适用场景   | 多租户、大型后台、权限由配置中心统一管理的系统                                                                                   |
| 代表实现   | vben `backend` 模式（`pageMap` + `layoutMap` 转换）                                                                              |
| 关键实现点 | 组件路径协议要版本化；映射不到的组件给统一 fallback 页；菜单接口要和权限接口同源，避免"菜单可见但接口 403"                       |
| 当前结论   | 后续扩展（等后端菜单协议稳定）                                                                                                   |

### 7.5 混合路由

前端静态路由和后端动态路由同时保留，按路由 `name` 合并，冲突时以后端元信息为准。

```text
核心页（登录、工作台、个人中心）：前端固定
业务页：后端菜单表下发
合并：byName merge，后端 meta 覆盖前端 meta
```

| 项目     | 说明                                                                                 |
| -------- | ------------------------------------------------------------------------------------ |
| 优点     | 核心页面前端可控（不怕后端菜单表出错导致整站不可用）；业务菜单后端可配置；扩展性最强 |
| 缺点     | 合并规则、冲突规则、调试成本最高；出问题时要同时排查两个来源                         |
| 适用场景 | 平台型后台、插件化后台                                                               |
| 代表实现 | vben `mixed` 模式                                                                    |
| 当前结论 | 远期可考虑                                                                           |

### 7.6 文件式路由（约定式）

路由不手写，由构建工具从 `pages/` 目录结构生成。这是"路由从哪来"的维度，权限控制仍需叠加 7.2–7.5 之一。

| 项目     | 说明                                                                                                   |
| -------- | ------------------------------------------------------------------------------------------------------ |
| 优点     | 零路由配置；文件即路由，重构目录即重构路由                                                             |
| 缺点     | meta 需要写在页面文件里（`definePage`）或额外配置；权限声明分散到各页面文件，失去 routes.ts 的集中总览 |
| 适用场景 | 内容型站点（Nuxt）、页面多且权限简单的项目                                                             |
| 代表实现 | unplugin-vue-router、Nuxt 约定式路由                                                                   |
| 当前结论 | 不采用（后台管理更看重权限集中声明）                                                                   |

### 选型速查

```text
页面 < 5、一次性        -> 7.1 也无妨
中小后台、权限在演进    -> 7.2（当前）
需要 404 隐藏路由       -> 7.3
改权限/菜单不发版       -> 7.4
平台化、插件化          -> 7.5
```

## 8. 当前方案分析

### 8.1 做对的点（对照最佳实践）

| 检查点                                                 | 现状                 |
| ------------------------------------------------------ | -------------------- |
| 守卫用返回值导航，不用已废弃的 `next()`                | ✅                   |
| 异步会话恢复正确 `await`，无竞态                       | ✅                   |
| 跳 Login/403 前判断目标是否已是自己，无重定向死循环    | ✅                   |
| `to.matched.every()` 逐层校验父子路由，不只查叶子      | ✅（很多项目漏这点） |
| 菜单与守卫共用 `access.ts` 同一判定函数，单一权限入口  | ✅                   |
| 权限码来自 contract 共享常量，前后端同源               | ✅                   |
| 前端权限只做 UX，后端 `requirePermission` 强制兜底     | ✅（最重要的一条）   |
| 按钮级权限有独立指令（`v-permission`），不散落 if 判断 | ✅                   |

### 8.2 真实的短板

| 短板                  | 说明                                                                                                                   | 严重程度 |
| --------------------- | ---------------------------------------------------------------------------------------------------------------------- | -------- |
| 403 暴露路径存在性    | 无权限用户直达 `/system/roles` 得到 403 而非 404，等于告知该路径存在。内部后台无所谓；安全敏感场景需要 7.3 的 404 语义 | 低       |
| ~~权限双轨并存~~      | `meta.roles` 已随迁移 022 删除，路由判定只剩 `permission` 单轨，见第 9 节                                              | 已消除   |
| routes.ts 单文件      | 当前约 400 行可接受；页面继续增长应拆成 `routes/modules/*`                                                             | 低       |
| bundle 含全部路由声明 | 懒加载下泄露的只是路径字符串，页面代码本身不会提前加载                                                                 | 低       |

### 8.3 结论

方案 7.2 对本项目是正确选择，不是妥协：权限模型刚跑通（迁移 019/020）、多数业务页还是占位页，此时上 7.3/7.4 是提前支付复杂度。触发升级的信号：

- 需要"改权限/菜单不发版" -> 升级 7.4
- 需要 404 隐藏无权限路由 -> 升级 7.3

在信号出现之前不需要更换判定架构。

## 9. 权限字段迁移状态

"迁移完成"指的是：**所有业务路由的访问判定从 `meta.roles`（角色粗粒度）迁到 `meta.permission`（权限码细粒度），`roles` 字段从 meta 类型中整体删除**。

每迁移一个路由需要三件事配套（权限模型换代后，权限码收敛到菜单表）：

```text
1. 后端：menus 表新增该页面/按钮节点（auth_code）+ 迁移脚本给角色种子绑定 role_menus
2. 契约：contracts/admin-api PERMISSION_CODES 增加常量
3. 前端：该路由 meta 删除 roles/permissions，改为 permission: PERMISSION_CODES.xxx
```

当前进度：**业务路由迁移已全部完成**（假业务占位路由已随菜单清理删除）。

| 路由            | 判定方式                         | 状态      |
| --------------- | -------------------------------- | --------- |
| `/system/users` | `permission`（system:user:view） | ✅ 已迁移 |
| `/system/roles` | `permission`（system:role:view） | ✅ 已迁移 |
| `/system/menus` | `permission`（system:menu:view） | ✅ 已迁移 |
| `/system/depts` | `permission`（system:dept:view） | ✅ 已迁移 |

`meta.roles` 与 `src/auth/rbac.ts` 已随迁移 022（删除 `users.role / users.admin_level`）整体移除。布局级路由（概览 `/dashboard`、系统管理容器 `/system`）不声明权限码，登录即可访问；系统管理目录在子项全被权限过滤后由菜单层整体隐藏。权限码全集 16 个：`system:{user,role,menu,dept}:{view,create,update,delete}`，与迁移 021 的菜单种子一一对应。

## 10. 参考项目方案对比

`admin-backend-2/apps/page-new` 参考项目的权限路由方案更完整，核心结构如下：

| 层级     | 参考项目做法                                                 |
| -------- | ------------------------------------------------------------ |
| 基础路由 | 登录、根布局、404 等作为 core routes，不进入权限生成流程     |
| 权限路由 | 业务路由从 `routes/modules` 收集，作为 access routes         |
| 权限生成 | 通过 `generateAccessible()` 按 `frontend/backend/mixed` 生成 |
| 路由注入 | 生成后动态加入根路由，再生成菜单                             |
| 菜单生成 | 通过最终可访问路由生成菜单                                   |
| 按钮权限 | 通过 `v-access` 指令和 `AccessControl` 组件控制              |
| 权限状态 | 独立 `accessStore` 保存 token、权限码、可访问菜单和路由      |
| 用户状态 | 独立 `userStore` 保存用户信息、角色、首页路径                |

和当前项目的区别：

| 对比点   | 当前方案                                  | 参考项目方案                   |
| -------- | ----------------------------------------- | ------------------------------ |
| 路由来源 | 前端静态路由                              | 前端、后端、混合三种来源       |
| 路由注册 | 启动时注册完整路由                        | 登录后生成并动态注入可访问路由 |
| 权限字段 | `permission`（唯一判定）                  | `authority`、`accessCodes`     |
| 菜单来源 | 从静态 `appRoutes` 派生过滤               | 从最终动态路由生成             |
| 按钮权限 | `v-permission` 指令（后端下发权限码判定） | 已提供指令和组件               |
| 复杂度   | 低，适合当前阶段                          | 高，适合成熟模板和复杂后台     |

参考项目更可扩展，但复杂度也明显更高。当前先采用静态 `meta` 方案是为了让路由、菜单、权限闭环更直观，后续按第 11 节顺序演进即可，不需要一次性引入全部封装。
