# 路由系统：静态路由、守卫与 RBAC 菜单

## 1. 当前目标

`apps/page` 第一版采用“静态路由 + 前端 RBAC 过滤”的方式实现后台管理基础骨架。

这个方案属于传统后台管理中常见的 RBAC 前端落地方式：页面路由通过 `meta` 声明需要的角色或权限，路由守卫负责拦截，菜单根据同一套路由权限过滤。

它的重点不是“前端决定最终权限”，而是“前端表达入口权限和交互权限”。真正的数据访问、接口写入、数据范围控制仍然必须由后端校验。

这一层先解决四个问题：

| 问题             | 当前处理方式                                                 |
| ---------------- | ------------------------------------------------------------ |
| 页面在哪里声明   | 在 `src/router/routes.ts` 中集中声明                         |
| 菜单从哪里来     | 从 `appRoutes` 的路由 `meta` 派生                            |
| 未登录如何拦截   | 全局路由守卫跳转 `/login`，并保留 `redirect`                 |
| 无权限如何处理   | 全局路由守卫跳转 `/403`                                      |
| 角色权限如何表达 | 后端用户字段先映射为 `super / admin / user` 三种前端访问角色 |

## 2. 源码位置

| 文件                                   | 作用                                                     |
| -------------------------------------- | -------------------------------------------------------- |
| `src/router/routes.ts`                 | 声明登录页、业务页、错误页、菜单标题、图标、角色、权限码 |
| `src/router/types.ts`                  | 扩展 Vue Router 的 `RouteMeta` 类型                      |
| `src/router/guards.ts`                 | 注册全局路由守卫                                         |
| `src/router/access.ts`                 | 判断用户是否能访问当前路由                               |
| `src/router/menu.ts`                   | 根据路由和用户角色生成侧边栏菜单数据                     |
| `src/auth/rbac.ts`                     | 把后端用户字段映射为前端访问角色                         |
| `src/stores/auth.ts`                   | 保存当前 token、用户快照、前端访问角色                   |
| `src/views/error/*.vue`                | 403 / 404 / 500 错误页                                   |
| `src/views/FeaturePlaceholderView.vue` | 业务页面迁移前的路由占位页                               |

## 3. 路由元信息设计

当前路由 `meta` 主要使用这些字段：

| 字段           | 说明                                              |
| -------------- | ------------------------------------------------- |
| `title`        | 页面标题，也是菜单标题来源                        |
| `icon`         | 菜单图标标识，后续侧边栏组件再映射到 Element 图标 |
| `order`        | 同级菜单排序                                      |
| `requiresAuth` | 是否需要登录                                      |
| `guestOnly`    | 已登录用户访问时自动跳回后台                      |
| `hideInMenu`   | 是否隐藏在侧边栏中                                |
| `activeMenu`   | 隐藏详情页用于高亮对应菜单                        |
| `roles`        | 允许访问的前端角色：`super / admin / user`        |
| `permissions`  | 权限码，第一版先声明，按钮级权限阶段再使用        |

示例：

```ts
{
  path: "/system/users",
  name: "SystemUsers",
  meta: {
    title: "用户管理",
    roles: ["super", "admin"],
    permissions: ["system:users:manage"],
  },
}
```

## 4. RBAC 映射

后端当前用户字段中已有：

```text
role
admin_level
```

前端先映射为三种访问角色：

| 后端字段组合                            | 前端角色 |
| --------------------------------------- | -------- |
| `role = admin` 且 `admin_level = super` | `super`  |
| `role = admin`                          | `admin`  |
| 其它普通用户                            | `user`   |

这样做的原因是：登录页、文档和后端数据已经按 `Super / Admin / User` 三类账号组织，第一版不需要先引入复杂角色表和权限表，先把页面入口权限跑通。

## 5. 守卫流程

当前全局守卫顺序：

```text
进入路由
  -> 从 localStorage 恢复 token 和用户快照到 Pinia
  -> 已登录访问登录页：跳转到 redirect 或 /dashboard
  -> 未登录访问受保护页面：跳转 /login?redirect=目标地址
  -> 已登录但角色不满足：跳转 /403
  -> 其它情况放行
```

守卫里没有发接口请求。原因是第一版先使用本地登录结果中的用户快照，避免页面刷新时因为 profile 接口未稳定导致路由链路不确定。后续接入“登录态恢复 + 当前用户接口”时，可以在进入后台布局前补一次 profile 校验。

## 6. 菜单派生

菜单不再手写一份独立配置，而是从 `appRoutes` 派生：

```text
路由配置
  -> 按用户角色过滤
  -> 去掉 hideInMenu
  -> 按 order 排序
  -> 生成侧边栏菜单
```

这样可以保证：

| 好处             | 说明                                            |
| ---------------- | ----------------------------------------------- |
| 页面和菜单不分裂 | 新增页面时只维护路由元信息                      |
| 权限判断统一     | 路由守卫和菜单过滤都使用同一套角色判断          |
| 隐藏路由清晰     | 详情页、错误页可以保留路由能力，但不进入菜单    |
| 方便后续替换     | 后端动态路由稳定后，可以替换 `appRoutes` 数据源 |

## 7. 路由权限方案背景

后台管理项目常见的路由权限方案大致分为五类。

### 7.1 手写菜单 + 手写路由

这种方案把侧边栏菜单和页面路由分别写成两份配置。

| 项目     | 说明                                             |
| -------- | ------------------------------------------------ |
| 优点     | 初期最快，适合极小项目或一次性后台               |
| 缺点     | 菜单、路由、权限容易分裂；新增页面要维护多处配置 |
| 适用场景 | 原型、小工具、权限很少的内部页面                 |
| 当前结论 | 不采用                                           |

不采用的原因是当前项目需要复习完整后台管理主线，菜单、路由和权限必须保持同源。

### 7.2 静态路由 + `meta` 权限声明

这种方案提前在前端声明全部业务路由，通过路由 `meta` 标注角色、权限码、是否进菜单、排序等信息。

```ts
{
  path: "/system/users",
  meta: {
    title: "用户管理",
    roles: ["super", "admin"],
    permissions: ["system:users:manage"],
  },
}
```

| 项目     | 说明                                                 |
| -------- | ---------------------------------------------------- |
| 优点     | 易理解、易测试、开发体验好；页面文件和路由关系清晰   |
| 缺点     | 前端仍能看到路由声明；最终权限必须以后端接口校验为准 |
| 适用场景 | 中小型后台、学习项目、权限模型还在演进的项目         |
| 当前结论 | 当前采用                                             |

这就是项目 3 当前采用的方案。它属于传统 RBAC 在前端路由层的常见实践，但只能负责“能不能看到入口、能不能进入页面”，不能替代后端权限。

### 7.3 静态路由 + 前端角色过滤后动态注入

这种方案也由前端维护完整路由表，但不是一开始全部注册到 Vue Router。登录后根据用户角色过滤路由，再把可访问路由动态加入路由实例。

参考项目 `admin-backend-2/apps/page-new` 的 `frontend` 模式就是这一类。它使用 `meta.authority` 表达角色权限，然后通过 `generateRoutesByFrontend()` 过滤出可访问路由。

| 项目     | 说明                                                     |
| -------- | -------------------------------------------------------- |
| 优点     | 用户路由表更干净；菜单来自最终可访问路由；适合模板项目   |
| 缺点     | 首次进入后台需要生成路由；刷新、切换用户、重置路由更复杂 |
| 适用场景 | 通用后台模板、中大型前端项目                             |
| 当前结论 | 后续可升级                                               |

项目 3 当前没有动态注入路由，原因是第一版更重视学习粒度和可读性。等后台布局、用户管理、角色管理完成后，可以再升级到这种方式。

### 7.4 后端动态路由

这种方案由后端返回菜单和路由结构，前端只负责把后端的组件字符串映射为真实页面组件。

参考项目的 `backend` 模式就是这一类。它调用菜单接口，拿到类似 `component: "views/system/users/index"` 的数据，再用 `pageMap` 和 `layoutMap` 转换为真实路由。

| 项目     | 说明                                                   |
| -------- | ------------------------------------------------------ |
| 优点     | 权限配置集中在后端；运营或管理员可以通过菜单表控制入口 |
| 缺点     | 需要稳定的菜单表、权限表、组件路径协议和兜底策略       |
| 适用场景 | 多租户、大型后台、权限由后台配置中心统一管理的系统     |
| 当前结论 | 后续扩展                                               |

项目 3 暂不采用，是因为当前后端 RBAC 表结构和菜单协议还没稳定。过早引入会增加学习负担。

### 7.5 混合路由

这种方案同时保留前端静态路由和后端动态路由，再按路由 `name` 合并。

参考项目的 `mixed` 模式就是这一类。它先生成前端可访问路由和后端返回路由，然后通过路由 `name` 合并，冲突时以后端元信息为准。

| 项目     | 说明                                                   |
| -------- | ------------------------------------------------------ |
| 优点     | 核心页面可前端固定，业务菜单可后端配置；扩展性最强     |
| 缺点     | 合并规则、冲突规则、调试成本最高                       |
| 适用场景 | 平台型后台、插件化后台、需要后端配置菜单但保留核心页面 |
| 当前结论 | 远期可考虑                                             |

对项目 3 来说，混合模式更像最终成熟形态，不适合第一版马上实现。

## 8. 参考项目方案对比

`admin-backend-2/apps/page-new` 参考项目的权限路由方案更完整，核心结构如下：

| 层级     | 参考项目做法                                                 |
| -------- | ------------------------------------------------------------ |
| 基础路由 | 登录、根布局、404 等作为 core routes，不进入权限生成流程     |
| 权限路由 | 业务路由从 `routes/modules` 收集，作为 access routes         |
| 权限生成 | 通过 `generateAccessible()` 按 `frontend/backend/mixed` 生成 |
| 路由注入 | 生成后动态加入根路由，再生成菜单                             |
| 菜单生成 | 通过最终可访问路由生成菜单                                   |
| 按钮权限 | 通过 `v-access` 指令和 `AccessControl` 组件控制              |
| 权限状态 | 独立 `accessStore` 保存 token、权限码、可访问菜单和路由      |
| 用户状态 | 独立 `userStore` 保存用户信息、角色、首页路径                |

和当前项目 3 的区别：

| 对比点   | 项目 3 当前方案             | 参考项目方案                   |
| -------- | --------------------------- | ------------------------------ |
| 路由来源 | 前端静态路由                | 前端、后端、混合三种来源       |
| 路由注册 | 启动时注册完整路由          | 登录后生成并动态注入可访问路由 |
| 权限字段 | `roles`、`permissions`      | `authority`、`accessCodes`     |
| 菜单来源 | 从静态 `appRoutes` 派生过滤 | 从最终动态路由生成             |
| 按钮权限 | 只预留权限码                | 已提供指令和组件               |
| 复杂度   | 低，适合学习第一版          | 高，适合成熟模板和复杂后台     |

参考项目更合理、更可扩展，但它的复杂度也明显更高。项目 3 当前阶段先采用静态 `meta` 方案，是为了让路由、菜单、权限闭环更直观。后续可以按参考项目逐步演进，而不是一次性引入全部封装。

## 10. 后续任务

下一步可以开始做后台布局：

```text
AppShell
  -> SidebarMenu 使用 createMenuItems(appRoutes, user)
  -> Topbar 保留通知 / 主题 / 用户
  -> Content 渲染 RouterView
```

布局完成后，再把 `FeaturePlaceholderView` 逐步替换为真实业务页面。

后续路由权限的推荐演进顺序：

```text
当前：静态路由 + meta 角色过滤
  -> 增加按钮级权限组件 / 指令
  -> 登录态恢复时拉取当前用户和权限码
  -> 引入前端动态注入可访问路由
  -> 后端 RBAC 表稳定后接入后端动态菜单
  -> 需要平台化能力时再考虑 mixed 混合模式
```
