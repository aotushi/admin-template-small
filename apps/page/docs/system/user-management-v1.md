# 用户管理模块 v1

## 模块定位

用户管理是当前后台管理模板的第一个真实业务页面，也是 RBAC 主线的入口页面。

第一版先完成“可查看、可筛选、可分页”的列表能力，不在同一步里强行补齐新增、编辑、删除、角色绑定。这样做的原因是：当前项目还处在后台管理模板骨架阶段，先把请求层、服务端状态、路由权限、页面容器和表格模式跑通，比一次性堆满 CRUD 更适合学习和后续扩展。

本模块遵守当前前端迁移规则：旧项目只提供用户数据、接口逻辑、权限逻辑和业务动作范围；最终 UI 不沿用旧项目，而是按 Vben 用户列表页的布局、查询区、表格工具栏和面包屑形态重新实现。

## 当前已实现

| 能力     | 当前实现                                                                                                  |
| -------- | --------------------------------------------------------------------------------------------------------- |
| 列表接口 | `GET /admin/api/users/list`                                                                               |
| 请求层   | `src/api/modules/users.ts` 通过统一 Axios 客户端请求后端                                                  |
| 查询状态 | `src/queries/users.ts` 使用 Pinia Colada 管理 query 状态                                                  |
| 页面入口 | `src/router/routes.ts` 中的 `/system/users`                                                               |
| 页面骨架 | 参考 Vben 列表页，采用左侧部门树 + 右侧查询表单 + 表格工作台                                              |
| 搜索筛选 | 支持用户名、用户 ID、状态、备注、创建时间筛选                                                             |
| 分页     | 当前使用前端分页，后续数据量变大后切换为后端分页                                                          |
| 列表选择 | 保留选择列和选中反馈，暂不接危险批量操作                                                                  |
| 权限承接 | 路由 `meta.permission` 权限码判定（`system:user:view`，后端下发），详见 `docs/router/route-guard-rbac.md` |
| CRUD     | 新增、编辑、删除已接入真实 mutation（Pinia Colada + query 失效刷新），删除带二次确认                      |
| 角色分配 | 编辑/新增弹窗内的角色下拉，仅 `super` 可用；后端同步写入 `user_roles` 绑定                                |
| 部门分配 | 弹窗内部门树选择（仅末级部门可选，超级管理员不归属部门，与后端校验一致）；左侧部门树来自真实接口          |

## 设计边界

| 层级       | 职责                                                |
| ---------- | --------------------------------------------------- |
| API        | 只描述后端接口，不处理页面状态                      |
| Query      | 只管理服务端数据状态，包括加载、错误、刷新、缓存    |
| View       | 组合页面、筛选状态、分页状态和工具栏反馈            |
| Department | 只负责展示部门树与选中筛选，数据来自真实部门接口    |
| Search     | 只负责查询表单，不直接请求数据                      |
| Table      | 只负责展示当前页数据、选择状态、工具栏和分页事件    |
| Pure logic | 负责可测试的筛选、分页规则，不依赖 Vue 组件生命周期 |

## 为什么删除页面标题与统计卡片

上一版页面顶部有 `System`、`用户管理`、说明文字和四个统计卡片。这个形态更像普通仪表盘页面，不像 Vben 的后台列表页。

当前改造后，页面信息由顶部图标面包屑承担，主体区域只保留业务工作台：

| 区域     | 当前处理方式                               |
| -------- | ------------------------------------------ |
| 面包屑   | 图标 + 文字，由路由 meta 生成              |
| 页面标题 | 删除，不再和面包屑重复表达当前位置         |
| 统计卡片 | 删除，用户管理 v1 先聚焦列表和查询         |
| 左侧树   | 保留 Vben 列表页的部门树布局，真实数据     |
| 查询表单 | 使用标签左置、两行多列、操作靠右的业务表单 |
| 列表工具 | 新增、搜索显隐、刷新、全屏、密度入口先占位 |

## 为什么当前先用前端分页

当前后端 `/api/users/list` 返回的是当前账号可见的用户列表，并且后端已经按管理员数据范围做了过滤：

- `Super` 可以看到全部用户。
- `Admin` 只能看到自己创建的用户。
- `User` 没有系统管理菜单权限。

在学习阶段，前端分页能先让列表页完整跑通，减少后端接口调整带来的干扰。后续用户量变大后，应把筛选条件、页码、页大小下沉到后端，前端只保留查询参数和展示状态。

| 新增用户 | ✅ 已实现：mutation + `UserFormDialog` 弹窗表单                                                                         |

## 为什么保留选择列

选择列适合保留，但只应该先作为列表基础交互存在。

Vben 的表格体系把表格能力抽象成配置，选择列、分页、查询表单、工具栏都属于后台列表页的标准骨架。当前项目不使用 `@vben` 封装，但可以学习这种页面形态：

| 设计点   | 当前采纳方式                                         |
| -------- | ---------------------------------------------------- |
| 选择列   | 使用 Element Plus `selection` 列                     |
| 选中反馈 | 在列表工具栏显示已选择数量                           |
| 批量操作 | 暂不接入，避免在权限和二次确认未完成前暴露危险操作   |
| 后续扩展 | 后续用于批量授权、批量删除、批量状态变更等管理员操作 |

这个取舍更适合学习型项目：先看到后台列表页的完整形态，再逐步补齐 mutation、权限按钮、确认弹窗和审计规则。

## 学习价值

这个模块可以用来复习后台管理中的几个关键点：

1. 路由权限如何决定菜单是否展示。
2. 登录态如何影响接口请求。
3. Axios 负责传输和 token，Pinia Colada 负责查询状态。
4. 页面组件如何避免写成一个巨大文件。
5. 纯函数如何把业务规则变成可测试内容。
6. 列表页如何从“静态占位页”过渡到“真实业务页”。
