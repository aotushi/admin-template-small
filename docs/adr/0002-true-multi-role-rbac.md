# 真多角色 RBAC（用户与角色多对多）

## 背景

历史模型是"表结构多对多、业务单角色"：`user_roles` 表存在，但 `users` 表另有 `role` 冗余列，接口与前端都只读写单个角色码，自定义角色无法从用户表单分配。同时 access token 携带 `role_codes`，导致角色变更要等 token 过期才生效。

按"后台管理按业务最佳实践设计，旧方案不合理即修复或删除"的原则重审后，决定把多对多贯通到全链路。

## 决定

1. **归属唯一存于 `user_roles`**（PK `(user_id, role_id)`）：`users` 表的 `role`/`admin_level` 列删除（迁移 022）。
2. **全链路多角色**：用户列表返回 `roles: [{code, name}]` + `role_codes: string[]`；创建/更新接收 `roles: string[]`；写入用 `d1.batch`（DELETE + 逐条 INSERT）保证原子替换。
3. **权限合并规则**：多角色权限码取并集；数据范围取最宽（`all` > `dept` > `self`）。
4. **super 互斥**：`super` 不可与其他角色叠加（前后端同规则校验）；删除/停用防线改为"不能移除最后一个启用状态的 super"。
5. **JWT 瘦身**：token subject 只含 `{id, username}`；角色与权限每请求由 `authMiddleware` 从 D1 实时解析，角色变更即时生效。
6. **用户表单角色选项动态化**：从 `/api/roles` 加载（仅 super 拉取），自定义角色可直接分配，分配闭环打通。

## 已考虑并否决的替代方案

- **业务保持单角色，删掉 `user_roles` 简化为外键**：实现最简，但自定义角色叠加（如"子管理员 + 审计员"）无法表达，后台管理常见诉求被堵死，否决。
- **role_codes 继续签进 JWT**：省一次每请求查询，但角色回收要等 token 过期（最长 15 分钟窗口），与"实时权限"目标冲突，否决。

## 后果

- 每个受保护请求多一次 D1 角色解析查询（与既有权限码实时解析同查询链路，边际成本小）。
- 前端角色展示从单标签变为标签组；编辑对话框需合并 `/api/roles` 选项与目标用户已绑定角色（覆盖停用/自定义角色在非 super 视角的回显）。
- 数据范围取最宽意味着给用户叠加角色只会放大可见范围，收窄必须靠移除角色，分配时需留意。
